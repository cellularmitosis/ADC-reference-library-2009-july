Overview of Documentation Sets

The Xcode IDE includes a full-featured documentation viewer that lets you view the installed developer documentation. The documentation window of Xcode has provided integrated browsing and searching of Apple’s developer documentation since Xcode 1.0.

In Xcode 3.0, you can integrate documentation for your own products into the Xcode Documentation window, using documentation sets. By providing your documentation as a documentation set, users can take advantage of all the Xcode documentation-viewing features to browse and search for information in your documents.

This chapter introduces the Xcode Documentation window and describes how documentation sets appear in the user interface. It also describes the documentation-set bundle and provides an overview of how to construct a documentation set.

Documentation in Xcode

When the user launches Xcode, it searches for installed documentation sets. It displays the documentation sets that it finds in the documentation set list in the Documentation window.

Using the Xcode documentation viewer, users can:

  • Perform full-text and API searches on your documentation.

  • Easily browse your documentation.

  • Use the Option–double-click keyboard shortcut from the code editor to jump to documentation for a symbol.

  • View summary information for symbols in your documentation using the Research Assistant.

  • Automatically check for and download documentation updates.

All these features, which Apple takes advantage of for its own developer documentation, are available to you when you package your own documentation as a documentation set.

Browsing Installed Content

By selecting a documentation set and clicking the Browse button, users can see and navigate the contents of the documentation set in the browser view. When the user selects an item in this browser view, the documentation window displays the associated HTML-based content in the content pane, as shown in Figure 1-1.


Figure 1-1  Browsing documentation


Each item in the documentation set list or in the browser view is a single documentation node. To the user, a node may be a single document, a collection of documents, or a single HTML documentation page. Within the documentation set, a documentation node represents a file or a set of files. The table of contents of your documentation set is determined by the hierarchy of documentation nodes that you define when you create the documentation set, as described in “Configuring Documentation Sets.”

Searching Content

Users select a type of search from the search strip (which appears below the Documentation window toolbar during a search) and type their search text in the search field. Xcode searches the documentation indicated by the user’s current selection in the documentation set list; this can be one or more documentation sets.

Note: Users can further control the scope of their search with the other options in the search strip. Search options are not visible in Browse mode; users can show search options by clicking the Browse button again.

For full-text searches, Xcode displays a list of results and their relevance, as shown in Figure 1-2.


Figure 1-2  Full-text search


For API searches, Xcode displays a list of matching symbol names, as shown in Figure 1-3.


Figure 1-3  API search


Full-text search is a batch operation; API and title searches filter the results list as the user types. The user can see the HTML document associated with a search result by selecting the search result.

Viewing Symbol Information from the Text Editor

When working in the Xcode code editor, users can get additional information about the currently selected symbol by doing either of the following:

  • Option–double-clicking the symbol to bring up the Documentation window and load the reference documentation for that symbol.

  • Bringing up the Research Assistant, shown in Figure 1-4.

    Figure 1-4  The Research Assistant window

    The Research Assistant shows a reference summary—such as a summary description, availability, and so forth—for the symbol, while allowing the user to continue working in the code editor. The user can jump to the full documentation for the symbol at any time. For more information on providing additional information for symbols, see “Supporting API Lookup in Documentation Sets.”

The Documentation Set Bundle

A documentation set is a standard Mac OS X bundle. The structure of a documentation set follows the conventions described in Bundle Programming Guide. Therefore, a documentation set’s content, including documentation files, can be localized.

The basic structure of an installed documentation set bundle is similar to that shown in Figure 1-5.


Figure 1-5  The structure of an installed documentation set bundle


The documentation set bundle contains the following (for the structure of a localized documentation set bundle, see “Internationalizing Documentation Sets”):

  • An information property list (Info.plist) file. This file describes the documentation set bundle. Xcode uses this file to:

    • Display the documentation set name in the documentation set list in the documentation window

    • Display information about the availability of updates to the documentation set and optionally download and install those updates

  • HTML documentation files. These must be placed inside of the Documents directory.

  • Generated index files. The docSet.dsidx and docSet.skidx files are binary files that describe the symbols and documents in the documentation set. Xcode uses the information in these files to:

    • Display the documentation set’s contents in the browser view in the documentation window

    • Carry out full-text searches of the HTML documentation files

    • Associate symbol names with locations in the HTML documentation

To generate the index files, you must also include several XML metadata files in your documentation set bundle. These files, which describe the contents of the documentation set, are:

  • A Nodes.xml file. This file describes the structure of the documentation set and is used to create the table of contents that users see in the Documentation window. This file is required.

  • One or more Tokens.xml files. These files (known as tokens files) associate symbol names with locations in the documentation and are used to create the symbol index for a documentation set. Although optional, you must include a tokens file to support fast API lookup.

These metadata files must reside within the documentation set bundle when you index the documentation set.

Documentation Set Development Workflow

The basic steps involved in creating a documentation set are:

  1. Organize documentation files into the bundle structure.

    The first step to building a documentation set is to create the directory hierarchy of the documentation set bundle and populate it with your HTML documentation files.

  2. Describe the documentation set and its structure.

    The next step in building a documentation set is to create an Info.plist file and Nodes.xml file for the documentation set. The Info.plist file describes the overall characteristics of the documentation set, such as its name and version number. The Nodes.xml file describes the structure of the documentation set and provides its table of contents.

  3. Support API lookup.

    If you wish to support API lookup for your documentation sets, you must also include one or more Tokens.xml files.

  4. (Optional) Localize the documentation set.

    If you provide documentation in more than one language, you can localize all or part of the documentation set bundle.

  5. Index the documentation set.

    The docsetutil tool generates the full-text (docSet.skidx) and API (docSet.dsidx) indexes that Xcode uses to access your documentation.

  6. (Optional) Provide an RSS or Atom feed.

    You can take advantage of automatic documentation updates in Xcode by providing an RSS or Atom feed that publishes updated versions of your documentation set.

  7. Test, package, and distribute the documentation set.

    Once your documentation set is complete, you can test it in Xcode simply by placing it in one of the standard documentation locations. You can also use the docsetutil tool to test the generated index files. After you have made sure that the documentation set and its indexes are complete and correct, you can use docsetutil to package the documentation set bundle as an XAR archive for distribution.
