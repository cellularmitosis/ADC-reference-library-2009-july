Class Description

NotQualifier is a subclass of Qualifier that contains a single qualifier. When a NotQualifier object is evaluated, it returns the inverse of the result obtained by evaluating the qualifier it contains.

NotQualifier implements the QualifierEvaluation interface, which defines the method evaluateWithObject for in-memory evaluation. When a NotQualifier object receives an evaluateWithObject message, it evaluates the given object to determine if it satisfies the qualifier criteria.

You can generate SQL code for a NotQualifier using the SQLExpression static method sqlStringForNegatedQualifier.

Creates and returns a new NotQualifier

If aQualifier is specified, it is used as the qualifier. For example, the following code excerpt constructs a qualifier, baseQual, and uses it to initialize a NotQualifier, negQual. The NotQualifier negQual is then used to filter an in-memory array. The code excerpt returns an array of Guest objects whose lastName properties do not have the same value as the lastName property of the guest's sponsoring member (this example is based on the Rentals sample database). In other words, the NotQualifier negQual inverts the effects of baseQual.

ImmutableVector guests /* Assume this exists */
Qualifier baseQual;
NotQualifier negQual;

baseQual = new Qualifier("lastName = member.lastName");
negQual = new NotQualifier(baseQual);
return Qualifier.filteredVectorWithQualifier(guests, negQual);

If aCoder is provided, the new NotQualifier is decoded from aCoder. This constructor, together with encodeWithCoder, provide the basis for archiving and distributing qualifiers. aCoder is specified when a NotQualifier is being read from an archive file; you never specify this constructor directly. For more information, see the next.util.Coding interface specification in the Foundation Framework Reference.

QualifierEvaluation interface

public boolean evaluateWithObject(java.lang.Object anObject)

Returns true if the object anObject satisfies the NotQualifier, false otherwise. This method can throw one of several possible exceptions if an error occurs. If your application allows users to construct arbitrary qualifiers (such as through a user interface), you may want to write code to catch any exceptions and properly respond to errors (for example, by displaying a panel saying that the user typed a poorly formed qualifier).
